# 项目约定

> 这是一份项目级长期约定文档，用于统一代码修改、接口变更、配置同步、缓存处理和文档维护的基本规则。
>
> 原则上，任何新增功能、重构、修复和优化，都应先检查本文件，再进入对应模块文档和代码实现。

## 1. 适用范围

本约定适用于以下范围：

- 后端 Java / SpringBoot / MySQL 相关代码
- 前端 Vue2 / ElementUI 管理端代码
- Flutter 移动端代码
- 数据库结构、配置项、缓存键、接口字段、文档说明

## 2. 项目基本事实

- 项目类型：投资理财后台系统
- 后端框架：RuoYi-Vue
- 后端语言：Java / SpringBoot
- 数据库：MySQL 8.4.8
- 管理端：Vue2 / ElementUI
- 移动端：Flutter
- 架构形态：单体架构，后台管理系统 + 移动端应用

## 3. 总体原则

1. 改动必须遵循“最小修改”原则，优先兼容现有代码风格和历史数据。
2. 不得擅自删除已确认的字段、配置、接口或业务规则。
3. 任何涉及用户字段、配置项、缓存键、接口返回结构的改动，都必须跨层同步。
4. 任何写库成功后的缓存读取链路，都必须保证先写库、后失效。
5. 涉及移动端展示的字段变更，必须同步 Flutter 模型和页面逻辑。
6. 涉及后台配置的新增项，必须同步后端、数据库、管理端、移动端四个位置。
7. 所有 SQL 变更应当可重复执行，并兼容旧数据。

## 4. 双币种硬规则

1. 币种仅包含 `CNY` 和 `USD`。
2. 系统基准货币固定为 `CNY`。
3. 所有资产、投资、充值、收益和统计口径，必须先统一换算为 `CNY` 再汇总。
4. 历史订单必须保存真实币种、原始金额、汇率快照和基准金额。
5. 货币兑换不做自动兑换，跨币种资金使用必须显式设计兑换流程。
6. 用户钱包按 `user_id + currency` 隔离，避免单钱包混币。

## 5. 配置同步规则

当新增或修改 `app.*` 配置项时，必须同时处理以下位置：

1. 后端 `AppConfigController`
2. 数据库 `sys_config`
3. Flutter `AppBootstrapConfigData`
4. `ruoyi-ui` 配置管理页

### 5.1 后端要求

- 配置键必须以 `app.` 开头。
- 后端读取逻辑应使用统一的 bootstrap/配置读取方式。
- 配置列表展示与默认值定义必须同步更新。

### 5.2 数据库要求

- 使用幂等方式写入配置项，避免重复执行报错。
- 配置说明、默认值和配置键必须保持一致。

### 5.3 前端要求

- Flutter 侧必须补齐字段、默认值、序列化、反序列化和访问方法。
- Vue 管理页必须补齐配置定义、输入控件、解析逻辑和保存逻辑。

## 6. 用户字段同步规则

当 `sys_user` 字段发生变化时，必须同时检查：

1. `SysUserMapper.xml`
2. `SysLoginController.getInfo`
3. `SysProfileController`
4. Flutter 用户资料模型
5. 相关页面展示逻辑

### 6.1 约束

- `getInfo` 必须返回 APP 侧需要的基础资料和派生状态。
- 后台完整权限树不应被 APP 轻量链路强行承载。
- 个人资料页只允许修改明确开放的字段，不能回写系统状态字段。

## 7. 缓存与失效规则

1. 先写库成功，再清理缓存。
2. 用户资料、实名状态、头像、登录信息等变化后，必须失效相关用户缓存。
3. 配置写操作成功后，必须执行 `resetConfigCache()`。
4. 不要只更新单个缓存 key 造成局部脏读。
5. 用户查询统一走 `ISysUserService`，不要在控制器层直接查表。

## 8. 登录与注册链路

1. APP 登录与注册保持轻量，不承载后台完整权限树。
2. 登录成功先保存 token，再异步刷新个人资料。
3. 注册成功直接签发 token，实现自动登录。
4. `getInfo` 仅负责补齐基础资料，不承担后台完整权限职责。
5. 用户资料更新应尽量避免联表和重查询。

## 9. 文档维护规则

项目文档分三层使用：

1. `docs/README.md`
2. `docs/project_context.md`
3. `docs/module_registry.md`
4. `docs/module_registry_login_register.md`
5. `docs/workflow_map.md`

### 9.1 维护原则

- 改代码之前先看对应模块文档。
- 改完代码后，若业务口径变化，必须回写文档。
- 若新增接口、字段、缓存、配置或页面行为，必须同步更新对应说明。

## 10. 变更流程

建议按以下顺序执行：

1. 确认变更属于哪个模块。
2. 阅读该模块文档和相关代码。
3. 识别受影响的后端、前端、数据库、缓存和配置项。
4. 先做最小改动，再补齐联动修改。
5. 编译、运行或回归验证。
6. 回写文档。

## 11. 变更检查清单

### 11.1 改接口

- 后端 Controller
- Service
- Mapper
- Flutter API
- 文档说明

### 11.2 改字段

- 数据库表结构
- 后端实体与 Mapper
- `getInfo` 返回结构
- Flutter 模型
- 页面展示与缓存失效

### 11.3 改配置

- `AppConfigController`
- `sys_config`
- Flutter 配置模型
- `ruoyi-ui` 配置页
- 文档说明

### 11.4 改缓存

- 写库后再清缓存
- 缓存 key 是否需要统一命名
- 是否需要补充失效入口
- 是否会影响多端读取

### 11.5 改 Flutter 页面

- 页面文案是否同步中英文
- 字段展示是否与后端返回一致
- 币种和金额显示是否遵循当前业务模式
- 本地缓存与服务端刷新逻辑是否一致

## 12. 推荐优先级

如果同时存在多个来源冲突，优先级建议如下：

1. 当前代码实现
2. `docs/project_context.md`
3. 对应模块文档
4. `docs/API_DOCUMENT.md`

如果文档和代码不一致，先以代码现状为准，再判断是否需要回写文档。

## 13. 结语

这份文档的目标不是增加流程负担，而是减少后续修改时的上下文丢失和口径漂移。

任何涉及业务口径的变更，都应当先看文档，再动代码，最后把新事实写回文档。
